This year I encountered the need to run certain processes
in the background consistently without root acces and without
using systemd (SSH appears to have issues with user services).
Generally speaking, the way to do this without any overhead is
evident: use `crontab` and have scripts that manage processes
in the background.  

# The service script
The service script can be either simple, not accepting any
arguments and just checking whether the process is running and
if it isn't, starting the process; or complicated, resembling
in its design a simple rc.d service script with according parameters.  

# Simple, using `nohup`
Certain processes don't need any input, nor they need to be
stopped or restarted. One such case is using an amazing SOCKS5
proxy server called [microsocks](https://github.com/rofl0r/microsocks).
The server doesn't need any `stdin` data and simply works.  

So it isn't a big deal if the script would just check for a user process
with the defined name and if there's none, it will start a new one. One important
thing to consider with this approach is strict process name matching: since the
"daemon" script is named `microsocksd`, running `grep` without `-w` key will result
in it detecting itself (the script) present and not starting the proxy server:

```sh
#!/bin/sh
cmds=$(ps -o cmd)
if echo $cmds | grep -wq microsocks; then exit 0; fi
nohup ~/.bin/microsocks -p $MICROSOCKS_PORT >/dev/null 2>&1 &
exit 0
```

The script is then put into the local `crontab` (via `crontab -e`) to run at fixed
intervals (in my case it's 5 minutes, seems like a good start if you want to
reduce your PID consumption if your VPS hoster limits PID amount, which apparently
is a thing, _the more you know_): 

```crontab
*/5 * * * * /home/und/.bin/microsocksd
```

# Complicated, using `screen`
Well, it's complicated, duh.  

If the target process/application is more sensitive and there could be running multiple
processes with the same name, it is safer to use GNU `screen` and design the service
script to resemble a generic `rc.d` script. Such script should have `start`, `stop` and `restart`
commands to control general behaviour, `watch` to start the process if it is not running
currently, and may have `stuff` to well... stuff a string of text to the process' standard input:  

```sh
sh service.sh start
sh service.sh restart
sh service.sh stop
sh service.sh watch
sh service.sh stuff An input string of text
```

One good example of such approach is my Minecraft™ SMP, running on a VPS where I don't
have root access and user systemd services won't work since I can only connect to it
through an SSH session. The automation also includes regular restarts as Java Edition
servers tend to lose stability at high uptimes. The service script for my SMP is available [here](https://github.com/untodesu/nrmc/blob/main/service.sh).  

```crontab
*/5 * * * * /home/und/nrmc/service.sh watch
30 19 * * * /home/und/nrmc/service.sh stuff broadcast "Restarting in 30 minutes"
45 19 * * * /home/und/nrmc/service.sh stuff broadcast "Restarting in 15 minutes"
55 19 * * * /home/und/nrmc/service.sh stuff broadcast "Restarting in  5 minutes"
55 19 * * * /home/und/nrmc/service.sh stuff save-all
00 20 * * * /home/und/nrmc/service.sh restart
```
